## 一、Query Rewrite（多轮对话指代消解）

### 问题

当前 `ChatServiceImpl.chatStream` 直接将用户输入传给 `KbChatAssistant.stream()` 作为检索查询。在多轮对话中：

```
用户：微信支付有哪些功能？
AI：  微信支付支持扫码支付、JSAPI 支付...
用户：它安全吗？                  ← "它"指代"微信支付"，但 pgvector 搜的是"它安全吗"
```

pgvector HYBRID 无法处理代词指代，导致检索结果偏离、回答质量下降。

### 方案

在 `ChatServiceImpl.chatStream` 中，调用 AI Service **之前**增加 Query Rewrite 步骤：

```
用户消息 → [加载最近 N 条历史] → [LLM 改写] → 改写后的 query → KbChatAssistant.stream()
```

### 新增组件：`QueryRewriter`

```java
// 位置：com.fast.knowledge.langchain4j.QueryRewriter
public interface QueryRewriter {
    /**
     * 基于对话历史改写当前查询，消解指代、补全省略。
     * 若无需改写（如首轮对话），返回原消息。
     */
    String rewrite(Long sessionId, String currentMessage);
}
```

- **实现**：`LlmQueryRewriter` — 调用当前配置的 `ChatModel`，用轻量 System Prompt + 最近 5 轮历史 + 当前消息做改写
- **Prompt 设计**：仅做指代消解和省略补全，不改变用户意图、不添加额外信息
- **容错**：LLM 调用失败时 fallback 到原始消息，不阻塞对话

### 改动范围

| 文件 | 改动 |
|------|------|
| `langchain4j/QueryRewriter.java` | **新增** 接口 |
| `langchain4j/LlmQueryRewriter.java` | **新增** 实现 |
| `service/impl/ChatServiceImpl.java` | 注入 `QueryRewriter`，在 `chatStream` 中调用 `rewrite()` |
| `model/dto/ChatMessageRequest.java` | 无需改动（改写对前端透明） |
| `service/MetricsService.java` | 新增 `countQueryRewrite()` 计数器 |

### 验收标准

- [ ] 多轮对话中，代词指代被正确解析为完整查询
- [ ] 首轮对话（无历史）时，原始查询不变
- [ ] LLM 改写失败时，fallback 到原始查询，对话不中断
- [ ] 检索结果与改写后查询相关（人工抽检 10 条）

## 二、检索与 RAG 质量埋点

### 问题

当前 `MetricsService` 已定义 `timeSearch`、`countSearch`、`timeIndex`、`countLlmCall` 等方法，但实际使用不完整：

- `SearchServiceImpl.search()` 调用了 `countSearch`，但**未用 `timeSearch` 包裹实际检索耗时**
- `RagServiceImpl.ask()` **完全未使用** `MetricsService`
- `ChatServiceImpl.chatStream()` 的流式对话延迟、Token 用量无记录
- 缺少零结果查询计数（检索返回 0 hits）

### 方案

补全埋点，使 v2.0.0 的"RAG 运营面板"有数据基础：

#### 2.1 SearchServiceImpl — 检索延迟埋点

```java
// 当前
metricsService.countSearch(kb.getId());
Embedding queryEmbedding = embeddingModel.embed(request.getQuery()).content();
// ...

// 改为
metricsService.countSearch(kb.getId());
List<SearchHitVO> hits = metricsService.timeSearch(kb.getId(), () -> {
    // 完整检索逻辑
});
metricsService.countSearchHits(kb.getId(), hits.size());  // 新增：命中数分布
```

#### 2.2 RagServiceImpl — RAG 全链路埋点

```java
// ask() 方法中新增
metricsService.timeRag(kbId, () -> { /* retrieve + generate */ });
metricsService.countLlmCall(model, "rag");
```

#### 2.3 ChatServiceImpl — 流式对话埋点

- 记录每次对话的 LLM 调用次数（`countLlmCall`）
- 记录检索耗时（由 LangChain4j 内部触发，但可通过 `onRetrieved` 回调计时）

#### 2.4 MetricsService 新增指标

| 指标 | 类型 | 说明 |
|------|------|------|
| `kb.search.hits` | Counter (tag: kbId, range: 0/1-5/6-20/20+) | 检索命中数分布 |
| `kb.rag.latency` | Timer (tag: kbId) | RAG 全链路延迟（检索+生成） |
| `kb.chat.latency` | Timer (tag: kbId) | 流式对话首 token 延迟 |
| `kb.query.rewrite.count` | Counter | Query Rewrite 调用次数 |

### 改动范围

| 文件 | 改动 |
|------|------|
| `service/MetricsService.java` | 新增 `countSearchHits()`、`timeRag()`、`timeChat()`、`countQueryRewrite()` |
| `service/impl/SearchServiceImpl.java` | 用 `timeSearch` 包裹检索逻辑，记录命中数 |
| `service/impl/RagServiceImpl.java` | 用 `timeRag` 包裹 ask()，记录 LLM 调用 |
| `service/impl/ChatServiceImpl.java` | 记录对话首 token 延迟、LLM 调用 |

### 验收标准

- [ ] `/actuator/metrics` 可查询到新增指标
- [ ] `kb.search.hits` 按命中数区间正确分桶
- [ ] `kb.rag.latency` 包含检索和生成两段耗时
- [ ] 现有功能无回归（埋点不改变业务逻辑）

## 三、异步任务队列升级（轮询 → Redis Pub/Sub）

### 问题

当前 `DocumentIngestServiceImpl` 使用 `@Scheduled(fixedDelay = 30000)` 轮询：

```java
@Scheduled(fixedDelay = 30000)
public void pollPendingTasks() {
    List<IndexTask> tasks = indexTaskMapper.findPending(5);
    for (IndexTask task : tasks) {
        scheduleIndex(task.getDocumentId());
    }
    // retryable 同理
}
```

痛点：
1. **延迟**：文档上传后最多等 30 秒才开始索引
2. **空转**：空闲时每 30 秒查询一次 DB，浪费连接
3. **重试粗糙**：失败任务依赖轮询重试，无退避策略
4. **无死信**：反复失败的任务无限重试

### 方案

#### 3.1 主路径：Redis Pub/Sub 事件驱动

```
文档上传 → scheduleIndex(docId)
         → 发布 Redis 消息 "kb:index:queue" { "docId": 123 }
         → IndexMessageListener.onMessage()
         → @Async indexExecutor 执行 indexDocument(docId)
```

- 利用已有的 Redis（`CacheProvider` 已对接），零新依赖
- Spring Data Redis `RedisMessageListenerContainer` 管理订阅

#### 3.2 兜底：低频轮询 + 退避重试

```
pollPendingTasks()  间隔从 30s → 5min（仅兜底，catch Pub/Sub 丢失的消息）
retryable 检查      指数退避：1min → 2min → 4min → 8min（最多 3 次）
死信                重试 3 次后标记 DEAD，记录错误，不再重试
```

- `IndexTask.retryCount` 字段已存在，利用 `CacheProvider.increment()` 做退避计数器
- 死信任务通过 `IndexTaskController` 手动重试

#### 3.3 新增组件

**`IndexEventPublisher`**：
```java
// 位置：com.fast.knowledge.service.IndexEventPublisher
public class IndexEventPublisher {
    void publish(Long documentId);  // 发布索引事件到 Redis
}
```

**`IndexMessageListener`**：
```java
// 位置：com.fast.knowledge.config.IndexMessageListener
public class IndexMessageListener implements MessageListener {
    void onMessage(Message message, byte[] pattern);  // 消费事件，调用 scheduleIndex
}
```

**Redis Pub/Sub 配置**：
```java
// 位置：com.fast.knowledge.config.RedisPubSubConfig
// 配置 RedisMessageListenerContainer + ChannelTopic
```

### 改动范围

| 文件 | 改动 |
|------|------|
| `service/IndexEventPublisher.java` | **新增** |
| `config/IndexMessageListener.java` | **新增** |
| `config/RedisPubSubConfig.java` | **新增** — `RedisMessageListenerContainer` Bean |
| `service/impl/DocumentIngestServiceImpl.java` | 注入 `IndexEventPublisher`；`scheduleIndex` 末尾 publish；`pollPendingTasks` 间隔改为 5min |
| `service/IndexTaskService.java` | 重试逻辑增加退避策略；新增 `DEAD` 状态 |
| `mapper/IndexTaskMapper.java` | 新增 `findDead()` 查询；重试 SQL 增加 `retry_count` 和 `updated_at` 条件 |
| `model/entity/IndexTask.java` | 状态枚举增加 `DEAD` |
| `controller/IndexTaskController.java` | 新增 `POST /index-tasks/{id}/retry` 手动重试死信 |

### 状态机变更

```
PENDING → INDEXING → INDEXED
                  ↘ FAILED → (退避) → PENDING  [最多 3 次]
                  ↘ FAILED → (超 3 次) → DEAD  [需手动重试]
```

### 验收标准

- [ ] 文档上传后 5 秒内开始索引（Pub/Sub 触发）
- [ ] Redis 不可用时，5 分钟兜底轮询仍能正常调度
- [ ] 失败任务指数退避重试（1min → 2min → 4min）
- [ ] 重试 3 次失败后标记 DEAD，不再自动重试
- [ ] DEAD 任务可通过 API 手动重试
- [ ] 索引吞吐量不低于轮询模式（当前每天最多数千文档）

## 四、不在 v1.1.0 范围内

以下在讨论中提及但明确推迟：

- **Chat Pipeline 显式化 / 插件链**：LangChain4j 已提供足够抽象，v2.0.0 与 Wiki 双路召回一起做
- **GraphRAG**：增加运维复杂度（需 Neo4j），当前 HYBRID 检索满足需求
- **Python 文档解析微服务**：Java POI/PDFBox/Tika 对当前格式覆盖足够
- **多向量数据库支持**：pgvector 单库降低运维复杂度，符合中小企业定位
- **Langfuse 全链路追踪**：Micrometer + Actuator 对当前规模足够，v2.0.0 再评估

**本次所有改动对 API 零影响**：

- 任务队列：索引行为不变，仅触发方式从轮询改为 Pub/Sub

## 七、风险与回退

| 风险 | 缓解 |
|------|------|
| Query Rewrite LLM 调用增加延迟 | 用轻量 prompt + 短上下文，目标 <1s；失败时 fallback 不阻塞 |
| Redis Pub/Sub 消息丢失 | 保留 5min 低频轮询兜底 |
| 指标埋点增加内存开销 | Micrometer 内存开销极小（<1MB），且仅影响 `/actuator/metrics` 端点 |

所有改动均可通过 feature flag 快速关闭：
- `knowledge.query.rewrite.enabled=true/false`
- `knowledge.index.pubsub.enabled=true/false`
