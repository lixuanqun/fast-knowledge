# Fast Knowledge 产品说明

> **Fast Knowledge（快速知识库）** — 面向中小企业的开源私有化知识库：Docker 一键部署、LangChain4j RAG、**LLM 中立**、数据本地可控。

---

## 一、产品定位

Fast Knowledge 是一款**轻量级企业知识库**，专为「**每家企业独立部署一套、数据不出内网、无需 SaaS 多租户**」的场景设计。

| 维度 | 定位 |
|------|------|
| **目标用户** | 中小企业、团队、部门级知识管理 |
| **部署模式** | 单实例私有化（Single Instance），每企一套，非多租户 SaaS |
| **数据规模** | 万级文档、数百用户 |
| **隐私策略** | Privacy by Default：文档、向量、对话默认留在本地服务器 |
| **AI 策略** | **LLM 中立**：OpenAI 兼容协议；Ollama / DeepSeek / 智谱 / 百炼 / 火山 / OpenAI / 自定义 |
| **运维复杂度** | Docker Compose 一键依赖，渐进启用 ONNX / Ollama / Reranker |

### 设计原则

1. **Privacy by Default** — 默认本地 Embedding（ONNX）+ 可选本地 LLM；`LLM_ALLOW_EXTERNAL=false` 可禁止外连大模型
2. **Single Instance** — 单租户，轻量工作区 + 知识库 ACL，无多租户运维负担
3. **Unified Stack** — PostgreSQL 统一承载业务表与 LangChain4j 向量表（`kb_embeddings`）
4. **Docker First** — 本地 `docker compose`，生产 K8s，统一 PostgreSQL + Redis + MinIO
5. **LangChain4j Native** — 摄入、混合检索、RAG、流式对话、Rerank 均走官方组件

### 适用场景

- 企业内部制度、手册、技术文档的集中管理与检索
- 销售/客服团队的 FAQ 与产品资料问答
- 研发团队的接口文档、设计文档知识沉淀
- 对数据出境敏感、需完全内网部署的组织
- 希望低成本试用 RAG，后续平滑升级向量与模型能力的团队

### 不适用场景

- 十万级以上文档、千人以上并发的大规模企业搜索平台
- 需要复杂多租户、计费、SaaS 运营的平台型产品
- 完全不愿自建任何 AI 组件、且不接受本地 ONNX/Ollama 的场景

## 四、核心功能详述

### 4.1 用户与权限

| 功能 | 说明 |
|------|------|
| JWT 单点登录 | Bearer Token 认证，支持登出与 Token 黑名单 |
| 首次安装向导 | 默认 `admin / admin123`，首次登录强制修改密码并设置实例名称 |
| 角色体系 | `ADMIN`（管理员）、`USER`（普通用户） |
| 用户管理 | 管理员可创建、编辑、禁用用户，重置密码 |
| 审计日志 | 记录知识库创建等关键操作，概览页可查看近期审计 |

### 4.2 知识库管理

| 功能 | 说明 |
|------|------|
| 工作区 | 轻量工作区（`kb_workspace`）组织知识库，自动为用户创建默认工作区 |
| 知识库 CRUD | 创建、编辑、删除知识库，支持名称与描述 |
| 可见性 | `PRIVATE`（私有）/ `PUBLIC`（全实例可见） |
| 成员 ACL | 知识库级成员权限：`READ` / `WRITE` / `ADMIN` |
| 检索参数 | `searchTopK`：每库检索/RAG 返回条数 |

### 4.3 文档管理

| 功能 | 说明 |
|------|------|
| 多格式上传 | PDF、DOCX、TXT、MD、PPTX、XLSX、HTML（Apache Tika 解析） |
| 文档预览 | 在线预览文档内容 |
| 分块查看 | 查看文档索引后的段落分块与 token 计数 |
| 异步索引 | 上传后后台异步向量化 |
| 单文档/整库重建 | 支持重新索引 |
| 智能写文档保存 | AI 生成的 Markdown 可直接保存入库 |
| 单文件限制 | 最大 50MB |

### 4.4 索引与检索

| 功能 | 说明 |
|------|------|
| 智能分块 | 段落感知分块 + Markdown 标题识别 + jtokkit token 计数 |
| 混合检索 | PgVector HYBRID（向量 + 全文 RRF 融合） |
| 检索重排序 | 可选 ONNX / Cohere / Jina，先扩召回再精排 |
| 检索缓存 | 相同查询结果缓存 5 分钟（Redis） |
| 索引任务管理 | 查看待处理/失败/死信任务，支持重试与整库重建 |
| HNSW 向量索引 | 启动时自动创建，`m` / `ef_construction` 可调 |
| Pub/Sub 事件驱动 | Redis 发布订阅触发索引，轮询 5min 兜底 |
| 指数退避重试 | 失败任务 1min→2min→4min 退避，超 3 次进死信 |

### 4.5 AI 能力

| 模块 | 能力 | 交互方式 |
|------|------|----------|
| **智能检索** | 语义 + 关键词混合搜索，返回相关段落与来源 | 同步 API |
| **智能问答** | 基于知识库 RAG 单轮问答，附引用来源 | 同步 API |
| **智能对话** | 多轮 RAG 对话，SSE 流式输出，会话持久化 | SSE 流式 |
| **智能写文档** | 按主题/大纲/风格生成 Markdown，可引用知识库 | SSE 流式 |

AI 基于 **LangChain4j**，对接 OpenAI 兼容 API。RAG 严格基于参考资料回答，不足时明确说明。

多轮对话支持 **Query Rewrite**（指代消解），Embedding 结果缓存避免重复推理，SSE 流式对话记录首 token 延迟。

### 4.6 系统配置与大模型

| 功能 | 说明 |
|------|------|
| 大模型配置页 | 管理员 `/settings/llm`，选择预设或自定义端点 |
| 热更新 | 保存后 `LlmModelRegistry.refresh()`，无需重启 |
| 配置存储 | `kb_system_config`（`llm.provider` 等 key） |
| 配置优先级 | 数据库（UI）> 环境变量 > 预设默认 |
| 公开只读 | `GET /system/config` 展示当前 LLM 策略（不泄露 Key） |

### 4.7 运营概览

- 知识库数量、文档总数、已索引/失败文档统计
- 待处理索引任务数量
- 近期审计日志列表

## 五、可插拔技术特性

### 5.1 向量存储

| 配置 | 说明 |
|------|------|
| `knowledge.vector.provider` | 固定 `pgvector` |
| `search-mode: HYBRID` | tsvector + 向量，RRF 融合 |
| 表名 | 默认 `kb_embeddings`，LangChain4j 自动建表 |

### 5.2 Embedding

| 提供方 | 配置值 | 适用场景 |
|--------|--------|----------|
| **ONNX 本地**（默认） | `onnx` | 生产离线，`bge-small-zh-v1.5` |
| **Ollama** | `ollama` | 全本地栈演示 |
| **Hash** | `hash` | 开发联调（`minimal` profile） |

### 5.3 缓存与存储

| 组件 | 说明 |
|------|------|
| **Caffeine L1** | 本地内存缓存，热查询 <1ms，500 条 / 2min TTL |
| **Redis L2** | 分布式检索缓存、Token 黑名单、索引锁、Pub/Sub |
| **MinIO** | S3 兼容对象存储 |

### 5.4 Reranker（可选）

| 提供方 | 说明 |
|--------|------|
| **ONNX 本地**（推荐） | `bge-reranker-base`，数据不出服务器 |
| **Cohere / Jina** | 云端 API 备选 |

### 5.5 大模型（LLM 中立）

| 方式 | 说明 |
|------|------|
| **管理界面** | `/settings/llm`，保存立即生效 |
| **环境变量** | `LLM_PROVIDER` / `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL` |
| **外连控制** | `LLM_ALLOW_EXTERNAL=false` |

详见 [deployment/llm-providers.md](./deployment/llm-providers.md)。
